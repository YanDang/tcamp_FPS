# 虚幻引擎5：多人联机FPS游戏 Demo 技术实现报告

**GitHub 链接：** [https://github.com/YanDang/tcamp_FPS](https://github.com/YanDang/tcamp_FPS)  
**开发环境：** Unreal Engine 5.4+ (基于官方 First Person 模板)

## 一、 核心玩法实现：AI 敌人行为与战斗逻辑

为了实现单局游戏的核心对抗目标，项目中开发了具备基础追踪与攻击能力的 AI 敌人（NPC），并完善了双向的伤害计算及持续作战系统。

### 1.1 场景寻路与 AI 追踪移动
* **寻路网格体：** 在关卡场景中全面铺设了 `NavMeshBoundsVolume`（寻路网格体边界体积），并构建了导航网格，确保 AI 能够识别地形障碍。
* **追踪逻辑：** 在 AI 敌人的事件图表中，使用 `AI MoveTo` 节点。通过将场景中的玩家角色（Player Character）设置为目标 Actor，AI 会在定时器的驱动下，自动计算最短路径并持续向玩家发起冲锋。

### 1.2 官方模板 State Tree（状态树）运行报错排查与修复
* **缺陷表象与报错：** 游戏运行时，AI 敌人不主动攻击，且引擎后台疯狂输出蓝图运行时错误：`"尝试读取StateTreeTask_SenseEnemies_C中的(real)属性Controller时，结果为无"` 以及 `"无访问"` 报错。报错节点主要集中在 `Bind Event to On Target Perception Updated / Forgotten`。
* **排查过程与根因分析：** 经查，这是 UE5 官方模板中应用较新的 **状态树（State Tree）** 系统时存在的一个遗留 Bug。在 `StateTreeTask_SenseEnemies` 任务节点中，系统试图通过 `Controller` 属性去绑定 AI 视觉感知的更新事件。但该 `Controller` 变量在状态树的数据流中变成了空指针（None），导致感知事件绑定全部失败，AI 感知系统彻底瘫痪，无法向玩家发起攻击。
* **重构与修复方案（上下文参数重绑定）：**
  1. **定位状态树资产：** 在 AI 控制器蓝图的 `State Tree AI` 组件中，找到并打开当前正在使用的绿色 State Tree 资产文件。
  2. **确认上下文（Context）：** 在状态树编辑器的 Context（上下文）根设置中，确认存在变量类型为 `AI Controller`（或 `Controller`）的全局上下文变量。
  3. **重新绑定输入参数（关键修正）：** 在状态树图表中选中调用 `Sense Enemies` 的状态节点，在右侧细节面板的 Tasks（任务）列表中选中 `StateTreeTask_SenseEnemies`。在其 Parameters（参数）区域找到悬空的 `Controller` 变量，点击绑定图标，将其显式 **Bind to（绑定到）** `Context -> AI Controller`。此举完美补齐了断裂的数据链，彻底消除了空指针报错并重新激活了 AI 的攻击行为逻辑。

### 1.3 敌人的攻击逻辑与玩家受击
* **攻击判定：** 为 AI 敌人添加了碰撞组件作为攻击范围触发器。当玩家进入该触发范围时，AI 状态树判定成功并触发攻击动作。
* **伤害施加：** AI 攻击命中玩家后，调用引擎底层的 `Apply Damage` 节点，将特定的伤害数值传递给玩家的 `Event AnyDamage` 接收端，从而扣减玩家的当前生命值。

### 1.4 玩家击败敌人与受击反馈
* 玩家通过武器发射的投射物（Projectile）命中 AI 敌人时，同样触发 AI 的 `Event AnyDamage`。
* 当 AI 生命值归零（`HP <= 0`）时，会调用死亡逻辑，切断其 AI 控制器输入，后续交由网络同步模块处理实体动画与销毁。

### 1.5 AI 敌人的随机复活机制（进阶功能）
为了保证游戏单局内的持续对抗体验，项目中为 AI 敌人设计了死亡后的自动复活循环：
* **随机生成点筛选：** 在 AI 死亡并等待尸体销毁（Destroy Actor）的过程中，服务器端的 `GameMode` 会获取场景中预设的所有出生点（如获取所有的 `PlayerStart` 或 `TargetPoint` 类），并使用 `Random Array Item` 节点从数组中随机抽取一个坐标（Transform）。
* **动态生成（Spawn）：** 服务器调用 `SpawnActorFromClass`，在选定的随机坐标点重新生成一个全新的 AI 敌人实例。这一机制成功实现了战场中敌人的源源不断，极大提升了 Demo 的可玩性。

## 二、 多人联机架构与网络同步重构

本项目采用了 **Listen Server（监听服务器）** 架构。为了保证游戏逻辑与物理表现的安全与统一，系统严格遵循“**服务器权威（Server Authoritative）**”原则。所有关乎游戏核心状态均在服务器端执行，客户端仅负责输入上传与视觉表现。在基于官方模板进行联机改造的过程中，重点修复了以下底层同步缺陷：

### 2.1 玩家开火控制权上交与“鬼弹”修复
* **缺陷表象：** 客户端（2P）开火时，本地能看到子弹飞出，但命中服务器无伤害，且远端看不到本地开枪。
* **重构方案：** 斩断客户端本地私自生成子弹的逻辑。将增强输入系统（`IA_Shoot`）的触发事件重定向至具备 `Run on Server (Reliable)` 属性的自定义事件 `Server_StartFiring`。由服务器权威校验并代为扣动扳机，生成全网同步的真实子弹 Actor（勾选 `Replicates`），确保所有玩家看到的弹道与伤害判定完全一致。

### 2.2 玩家与 AI 死亡状态的跨网同步
* **缺陷表象：** 服务器击杀敌人或 2P 死亡时，仅服务器本地能看到布娃娃（Ragdoll）倒地效果，2P 客户端看到的尸体依然保持站立硬直（僵尸状态）。
* **重构方案：** 将数据结算与视觉表现彻底剥离。伤害判定与分数增加逻辑仅在服务器后台运行；而纯视觉节点（如 `Set Simulate Physics`）则被封装进全新的**多播自定义事件（NetMulticast）** 中。当血量归零时，服务器呼叫大喇叭，强制所有客户端本地同步渲染物理倒地效果。

### 2.3 玩家网格体双重渲染（第一/第三人称视口穿模）修复
* **缺陷表象：** 在多人联机中，玩家看向彼此的时候，会发现对方同时渲染了“第一人称的浮空手臂”和“第三人称的全身模型”，两个身体严重穿模重叠。
* **根因分析：** 标准 FPS 游戏中，角色蓝图通常包含两个骨骼网格体（Mesh）：一个是供玩家自己看的第一人称手臂（FirstPersonMesh），另一个是供其他玩家看的第三人称全身模型（ThirdPersonMesh）。如果未配置网络视口隔离，引擎默认会将两者同时渲染给所有客户端。
* **重构方案：** 利用引擎底层的渲染可见性（Visibility）权限进行隔离排他处理：
  1. 将第一人称手臂组件（FirstPersonMesh）及其绑定的武器属性勾选为 **`Only Owner See`（仅所有者可见）**，确保其他玩家看不到漂浮的手臂。
  2. 将第三人称全身组件（Mesh）勾选为 **`Owner No See`（所有者不可见）**，确保本地玩家的视口不会被自己的无头模型遮挡，但远端网络玩家可以清晰看到完整的第三人称持枪姿态。

## 三、 得分与游戏胜利机制开发

系统在服务器权威的 `GameMode` 与全网同步的 `GameState` 之间搭建了一套完备的终局判定流水线。

### 3.1 胜负评定数据流控
当 AI 被击杀或敌对玩家阵亡触发服务器的加分操作时，`BP_ShooterGameMode` 会获取并改写当前的队伍得分，并立即通过预设的临界条件函数进行规则审计：
设判定胜负的目标极限分数为 $S_{target}$，当前队伍得分数值为 $S_{current}$。若满足关系：
$$S_{current} \ge S_{target}$$
则判定当前队伍获胜，立刻进入比赛结算阶段。

### 3.2 终局状态广播与输入冻结
一旦胜负产生，服务器 `GameMode` 将执行以下终局操作：
1. **输入冻结：** 利用网络传导向全网所有玩家发送指令，调用 `Disable Input` 节点切断玩家控制器的输入映射，全面冻结开火、移动与视角旋转，防止终局后玩家继续行动。
2. **动量清空：** 通知玩家的 `CharacterMovement` 组件执行 `Stop Movement Immediately`，强行清空当前的物理惯性。
3. **结算 UI 展示：** 触发本地控制器的结算事件，在玩家屏幕上渲染“VICTORY（胜利）”或“DEFEAT（失败）”的结算界面，完成单局游戏的闭环。

## 四、 实际部署遗留问题与未来优化方向

在实际的网络环境真机测试与部署中，由于时间限制与引擎底层网络同步机制的复杂性，部分非核心体验问题暂时被搁置，这些问题明确了本项目后续的迭代与优化方向：

### 4.1 复杂武器切换的模型与动画错位（已知遗留）
* **现象描述：** 当客户端（2P）频繁切换武器（如手枪切至步枪）时，偶尔会出现枪械模型贴图未刷新，或角色依然使用手枪的骨骼动画姿态强行握持步枪的穿帮现象。
* **原因分析：** 可能是武器网络所有权（Owner）下发与客户端动画蓝图（AnimBP）更替之间存在微小的“网络时差（Race Condition）”。
* **未来优化：** 计划在 `OnRep_CurrentWeapon` 回调中引入代理事件，增加微小缓冲期以等待新武器材质下载完毕；改由客户端角色底层直接调用 `Set Actor Hidden In Game` 强制接管旧武器的视觉隐藏。

### 4.2 客户端 UI 的数据隔离与同步延迟（已知遗留）
* **现象描述：** 客户端（2P）在进入游戏初期，屏幕顶部的比分板可能无法立刻显示分数。
* **原因分析：** 客户端本地挂载 UI Widget 的时机可能早于 `GameState` 数据首次同步下发的时机。
* **未来优化：** 将 UI 的驱动机制从每帧刷新（Tick）优化为**“事件驱动机制（Event-Driven）”**。在 `GameState` 内利用 `RepNotify` 变量，仅在分数发生实质改变时，定向推送事件通知 UI 更新。

### 4.3 射击手感的本地预测补偿
目前架构为了追求绝对的数据安全，客户端开火完全等待服务器 RPC 响应，在延迟较高的网络下会产生“开枪手感粘滞”。未来计划引入“本地预测生成虚拟弹道与火花特效，后台等待服务器权威伤害判定”的解耦方案，以提升射击爽快感。
